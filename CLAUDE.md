# CLAUDE.md — MicroDNS

## Project Overview

MicroDNS is a Rust workspace replacing PowerDNS + pdnsloadbalancer with a single binary: authoritative DNS, recursive DNS, load balancing, DHCP (v4/v6/SLAAC), and IPAM. Instance-per-network federated topology where each subnet runs its own instance.

## Architecture

- **Container runtime**: stormdbase (process supervisor, SSH, logging, health probes, busybox commands)
- **Database**: redb (embedded, ACID, memory-mapped — no external DB needed)
- **DNS**: hickory-dns 0.24 (auth + recursor)
- **HTTP**: axum 0.8 (REST API + dashboard)
- **gRPC**: tonic 0.12
- **Async**: tokio 1.x
- **Config**: TOML for bootstrap only, all runtime config stored in redb and managed via REST API
- **Messaging**: NATS (optional, for federation and event publishing)

## Build & Deploy

**Local machine is ARM64** (Apple Silicon Mac).

```bash
# One-step build + deploy (cross-compile, container build, push)
scripts/build-and-push.sh

# Manual steps:
cargo build --release --target aarch64-unknown-linux-musl
podman build --tls-verify=false --platform linux/arm64 -f Dockerfile.scratch -t registry.gt.lo:5000/microdns:edge .
podman push --tls-verify=false registry.gt.lo:5000/microdns:edge
```

- mkube auto-deploys — watches the registry, pulls new images (~2 min)
- Rust 1.88+ required (time-core needs edition2024)
- `.dockerignore` must have `target/` to avoid 4GB+ build context
- Base image: `registry.gt.lo:5000/stormdbase:latest` (not scratch)

## Key Design Decisions

### DNS NOERROR vs NXDOMAIN
When a name exists but has no records of the queried type (e.g., AAAA query for a name with only A records), microdns returns **NOERROR with empty answer**, not NXDOMAIN. This is critical because systemd-resolved does parallel A+AAAA lookups and NXDOMAIN on either poisons the entire result.

### DHCP Option 15/119 Handling
When `domain_search` (option 119) is configured on a pool, option 15 (domain name) is suppressed. Option 15 causes systemd-resolved to scope the DNS server to a single domain, breaking cross-network resolution.

### Database-Driven Config
All runtime config (pools, reservations, forwarders) is stored in redb and managed via REST API. TOML is only used for initial bootstrap. Changes take effect immediately — redb is memory-mapped, reads are free, no reload signals needed.

### DHCP Reservation Inheritance
Reservation fields are all `Option<T>`. When not set on the reservation, pool defaults are used. A minimal reservation only needs `mac`, `ip`, and `hostname`.

### Graceful Shutdown
API server has a 5s shutdown timeout, main binary has 8s overall timeout. Prevents container restart loops from long-lived WebSocket/SSE connections.

### Forward-with-Fallback
Each instance forwards queries for peer zones to the peer's DNS server. If the peer is down, serves a local copy of the zone data (non-authoritative, AA=0).

## Recent Fixes (2026-03-20)

- **NXDOMAIN → NOERROR**: Fixed parallel A+AAAA lookup poisoning for systemd-resolved
- **Option 15 suppression**: Prevents DNS scoping when domain_search is configured
- **Shutdown timeout**: 8s bounded shutdown prevents container restart loops
- **Cross-network forwarding**: All instances forward to all peer zones including reverse and utility zones
- **stormdbase migration**: Container base switched from scratch to stormdbase for process supervision, SSH, health probes
